# Переход с Rules 3.x / RC.5 на историческую реализацию RC.6

Эта инструкция описывает редакцию репозитория
[`193a4a2`](https://github.com/jsonspecs/rules/commit/193a4a2), которая реализовывала
RC.6 до создания тега выпуска `v4.0.0`. Опубликованный пакет Rules 4.0.0 работает с
RC.7; актуальный переход описан в [MIGRATION_RC7_RU.md](MIGRATION_RC7_RU.md). В
редакции RC.6 не было переключателя или режима совместимости с RC.5.

## Снэпшот

1. Измените `specVersion` на `"1.0.0-rc.6"`.
2. Пересоберите снэпшот и заново вычислите `sourceHash` для полного снэпшота RC.6.
3. Для точного воспроизведения исторического поведения используйте редакцию Rules
   `193a4a2`. Не устанавливайте опубликованный пакет `@jsonspecs/rules@4.0.0` для
   снэпшота RC.6: этот пакет работает с RC.7 и отклоняет RC.6.

Снэпшот RC.5 отклоняется с кодом `UNSUPPORTED_SPEC_VERSION`, даже если форма его
артефактов в остальном допустима.

## Поведение `[*]`

RC.5 выбирал существующие конечные пути из плоской проекции. RC.6 сначала перечисляет
реальные элементы массива из вложенного `payload`, а затем определяет конечный путь
каждого кандидата как присутствующий или отсутствующий.

Например, `items[*].sku` теперь создаёт одного кандидата на каждый реальный элемент.
Правило `not_empty` с `ALL + EACH` сообщает об отсутствующем дочернем поле как об
`items[1].sku` и не добавляет `actual` в такую ошибку. Оператор значения для того же
отсутствующего кандидата по-прежнему получает `SKIP` на уровне ядра.

При пересборке пакета проверьте следующие изменения:

- `matched` включает отсутствующих структурных кандидатов;
- отсутствие для оператора значения увеличивает `skipped`;
- популяция, целиком состоящая из `SKIP`, остаётся `SKIP` и не применяет `onEmpty`;
- `onEmpty` применяется только при полном отсутствии структурных кандидатов;
- отсутствие до следующего `[*]` не создаёт ветвь, потому что реальный следующий
  индекс неизвестен;
- непустой контейнер в конечном `items[*]` остаётся отсутствующим значением и не
  передаётся оператору.

Правила обязательности и значения должны оставаться раздельными. Для обязательного
дочернего поля каждого существующего элемента используйте оператор присутствия с
`aggregate: { mode: "ALL", onEmpty: "SKIP", issueMode: "EACH" }`. Проверку пустой
коллекции оставьте в отдельном существующем бизнес-правиле.

Полное поведение задано в закреплённой
[спецификации RC.6](https://github.com/jsonspecs/spec/blob/d75024047437ce0119a28c6ceda818eb79c4f302/SPEC_RU.md)
и её [руководстве по переходу](https://github.com/jsonspecs/spec/blob/d75024047437ce0119a28c6ceda818eb79c4f302/MIGRATION_RC6_RU.md).
